# 102 · BUG 修复工作流程

> **触发条件**：开发者提出"BUG、报错、异常、不对、修一下"等意图。
> **核心思想**：**先复现，后定位，再修复，最后回归。** 不能复现的 BUG 不应该被"修复"——大概率没修对。

---

## 步骤总览

```
(1) BUG 记录 + 复现  →  (2) 根因定位  →  (3) 修复方案 + 用户确认  →
(4) 修复实施  →  (5) 即时验证 + 同类排查  →  (6) 测试（进入 104）  →  (7) 审查（进入 103）  →  (8) 归档
```

---

## (1) BUG 记录 + 复现 ⚠️ 不能复现 = 不能修

### 做什么
- 把 BUG 写到记录文档里：`docs/03_问题记录/yyyymmdd_BUG_xxx问题描述.md`
- 模板：`docs/20_模板/BUG记录模板.md`
- 必须包含：
  - 现象描述（什么操作、什么结果、是否截图）
  - **复现步骤**（编号化，可被他人按此重现）
  - 期望行为
  - 实际行为
  - 影响范围（是个例还是普遍？哪些角色受影响？）
  - 出现频率、首次出现时间、相关近期变更（如有）

### 复现卡点
- ✅ 能稳定复现 → 进入根因定位
- ⚠️ 偶发不能稳定复现 → **先列出怀疑场景**，加日志/埋点、用户确认后再继续；不要靠"猜"修
- ❌ 完全不能复现且无证据 → **暂停工作流**，回去与开发者要更多信息（截图、日志、时间点、用户 ID）

---

## (2) 根因定位

### 思考路径（按顺序排除）
1. **数据问题 还是 逻辑问题？**
   - 一条数据出问题：八成是数据问题（脏数据 / 边界数据）
   - 所有数据出问题：八成是逻辑问题
2. **前端 还是 后端？**
   - 看 Network 面板：接口入参对不对、出参对不对、错误码是多少
   - 接口返回正确但显示错误 → 前端
   - 接口返回错误 → 后端
3. **单模块 还是 跨模块联动？**
   - 涉及消息队列、缓存、异步任务、定时任务时，跨模块的可能性大
4. **是否有相似代码同样有问题？**
   - grep 找出相同模式，**很多 BUG 不是孤例**

### 工具与手段
- 读代码 → 阅读相关函数完整调用链
- 看日志 → 失败时间点前后的日志
- 数据库查询 → 直接看出问题数据的状态
- 二分回溯 → 看最近改动（git log / git blame）
- 加临时日志 → 复现一次拿到运行时数据，**修完后必须删除临时日志**

### 产出
在 BUG 文档"根因分析"小节写清：
- [ ] 根因的**一句话结论**
- [ ] 触发条件（什么数据/什么角色/什么并发）
- [ ] 为什么之前没发现（代码逻辑、测试覆盖、上线流程）
- [ ] 是否还有相似代码可能有同样问题

---

## (3) 修复方案 + 用户确认 ⚠️ 卡点

### 修复策略选择
| 策略 | 适用 | 注意 |
|---|---|---|
| 直接修复 | 根因清晰、影响面可控 | 优先选 |
| 迂回修复 | 根因清晰但直接修代价大 | 必须在文档中说明"为什么不直接改" |
| 临时修复 + 长期方案 | 线上紧急救火 | 必须创建长期方案的 TODO 项 |

### 必须告诉开发者
- 根因结论
- 修复策略与具体改动
- 可能的副作用（包括正在使用此功能的用户体验变化）
- 是否需要**数据修复**（脏数据修正脚本）
- 影响范围大的 BUG，**主动建议在测试环境先验证**

### 卡点
- 影响范围大的 BUG，**必须**收到开发者明确确认才动手
- 涉及生产数据修复的，必须开发者给出明确的"可以执行 SQL"

---

## (4) 修复实施

### 红线
- ❌ 禁止修复时**顺手优化无关代码**——增加回归风险
- ❌ 禁止**用注释掉的方式"修"** —— 要删就删
- ❌ 禁止用 `try/catch` 吞掉错误当作修复——根因要解决
- ✓ 临时日志在提交前删除（或转为正式可观测性日志）

### 数据修复
- 写 SQL 修脏数据时，**先 SELECT 验证范围**，确认无误后再 UPDATE
- 数据修复 SQL 必须归档到 BUG 文档"数据修复脚本"小节
- 数据修复在事务里执行，影响行数超预期立刻回滚

---

## (5) 即时验证 + 同类排查

### 验证四步
1. **按原复现步骤再走一遍** → 应该看不到 BUG 了
2. **执行边界变种** → 临界值、空值、超长、并发等情况
3. **回归相邻功能** → 同模块的其他常用路径
4. **同模式 grep** → 如果根因是"忘了某种判断"，全仓搜相似代码，一并修

### 自检
- [ ] BUG 复现步骤已 100% 通过
- [ ] 同类代码已 grep 排查
- [ ] 临时日志/调试代码已清理
- [ ] 数据修复脚本已归档（如有）

## 注意事项

- **不要急** —— 看似简单的 BUG 经常有更深的根因，没看透就改是埋雷
- **不要恋战** —— 定位超过 1 小时没头绪，主动找开发者一起看，或先记录现状停下来
- **不要孤立看待** —— 同一根因可能在多处发生，grep 搜一遍永远不会错
- **不要相信"应该不会再发"** —— 加回归用例是硬性要求
- **临时修复必须有 TODO** —— 否则 100% 变成永久债务
- **复现不了的 BUG 不要瞎改** —— 改了也不知道修没修，可能制造新问题
