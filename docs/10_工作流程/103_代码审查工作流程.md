# 103 · 代码审查工作流程

> **触发条件**：开发者提出"审一下代码、Code Review、看看 PR"等意图；或工作流 100/101/102 完成开发后强制进入。
> **核心思想**：审查不是找拼写错误，是从**正确性、规范性、可维护性、安全性**四个维度系统化体检。

---

## 审查角色与时机

| 触发场景 | 审查者 | 重点 |
|---|---|---|
| AI 完成开发后自审 | AI（本工作流必跑） | 全维度自检 |
| 开发者复审 AI 产出 | 人 | 业务正确性、风格契合度 |
| Pull Request | 同伴 + AI | 跨视角对照 |

---

## 审查的四个维度

### 维度一：正确性（最高优先级）

#### 业务正确
- [ ] 是否完整实现了需求/方案的全部内容？逐条对照验收标准
- [ ] 边界条件处理：空值、null、空数组、零、负数、超大值、超长字符串
- [ ] 异常路径处理：网络超时、外部依赖失败、并发冲突、权限不足
- [ ] 业务规则是否一致：同一逻辑在多处出现时是否表达一致？

#### 代码正确
- [ ] 类型 / 空指针 / 除零 / 数组越界
- [ ] 死循环 / 死锁 / 无限递归
- [ ] 资源未释放（文件句柄、连接、订阅、监听器）
- [ ] 数值精度（浮点、金额、百分比、日期范围）

#### 数据正确
- [ ] 数据库事务边界：跨表写入是否在同一事务内？外部 HTTP/MQ 调用**是否被卷入事务**？（要避免）
- [ ] 软删除字段：所有相关查询是否带 `deleted = false`？
- [ ] 唯一约束：业务唯一性是否在 DB 层有索引保护？
- [ ] 并发安全：乐观锁 / 唯一索引 / 防重 token

---

### 维度二：规范性

对照 `docs/00_通用规范/` 逐项检查：

- [ ] **命名**：变量、函数、类、文件、目录是否符合规范？
- [ ] **代码风格**：缩进、空格、单文件长度、单函数长度、嵌套深度
- [ ] **注释**：注释是否在写"为什么"而不是"是什么"？无意义注释是否清理？
- [ ] **API 设计**：REST 路径、HTTP 方法、错误码、字段命名是否符合规范？
- [ ] **数据库**：表/字段命名、主键策略、索引、外键 / 软删除字段
- [ ] **错误处理与日志**：异常是否吞掉？日志级别是否正确？是否打印敏感信息？
- [ ] **Git 提交**：分支命名、提交粒度、提交信息

---

### 维度三：可维护性

- [ ] **单一职责**：函数/类是否做了太多事情？
- [ ] **重复代码**：相似代码块是否应该抽象？（注意：**三处以下重复不必抽象**）
- [ ] **过度设计**：是否引入了用不上的"灵活性"和"扩展点"？
- [ ] **魔法值**：硬编码字符串/数字是否应该提取常量？
- [ ] **依赖方向**：模块依赖是否符合分层架构？（如：业务层不能依赖控制器）
- [ ] **TODO/FIXME**：是否有遗留的临时代码、调试代码、`console.log`？

---

### 维度四：安全性

对照 `docs/00_通用规范/06_安全规范.md`：

- [ ] **认证**：未认证请求是否拒绝？token 校验是否实现？
- [ ] **授权**：功能权限是否在后端校验（不是只在前端隐藏）？数据权限（行级）是否在 SQL 层过滤？
- [ ] **越权**：构造其他用户 ID 是否能读到不该读的数据？导出接口是否绕过列表权限？
- [ ] **输入校验**：参数校验是否在后端做了？仅前端校验视为无效
- [ ] **注入**：SQL 拼接、命令拼接、模板拼接是否使用了参数化方式？
- [ ] **敏感数据**：密码、token、身份证、手机号在日志/响应中是否脱敏？
- [ ] **依赖安全**：新增的第三方库是否经过基本审查（来源、维护状态、license）？

## 审查的两种模式

### 模式 A：AI 自审（必跑）

AI 在工作流 100/101/102 的"代码自审"步骤进入本工作流。
执行：
1. 对照本文 §维度一~四 逐项过一遍
2. 缺陷分类：
   - 阻断（必修，否则视为未完成）
   - 重要（强烈建议修）
   - 建议（可选改进）
3. 输出**审查报告**到 `docs/04_代码审查/yyyymmdd_审查_xxx.md`（模板见 `docs/20_模板/代码审查报告模板.md`）
4. 阻断项**自己修完**再交付；重要项与开发者商量是否当前修

### 模式 B：人工审查 AI 产出

开发者审查 AI 代码时，AI 的角色是**配合解释而不是辩护**：
- 开发者指出问题 → AI 先理解再回应
- 不理解时反问澄清，不要假装理解
- 修改后**再次自审**整遍，不要只盯着被指出的一处

---

## 审查的红线

- ❌ **走过场审查**——只看了几个文件就 LGTM，无报告无证据
- ❌ **盲目套规范**——规范明显不适用本场景时应该例外说明，而不是硬套
- ❌ **审查者甩锅**——审过的代码出问题，审查者也有责任，不能"是开发者写的与我无关"
- ❌ **批评不给建议**——指出问题时要给出可行的修复方向

---

## 审查报告

### 产出位置
`docs/04_代码审查/yyyymmdd_审查_xxx.md`

### 必含内容
- 审查对象（PR 链接 / commit / 文件清单）
- 维度一/二/三/四的检查结果
- 缺陷清单（按阻断/重要/建议分类）
- 结论：✅ 通过 / ⚠️ 修复后通过 / ❌ 不通过
- 后续动作清单

---

## 自检（审查者用）

- [ ] 四个维度都检查过，**不是只看了一两个维度**
- [ ] 通读了核心改动（不只是看了 diff 摘要）
- [ ] 关键路径在脑中跑通了，包括异常分支
- [ ] 跨文件影响已确认（grep 搜被改动符号/接口/表）
- [ ] 审查报告已落盘

---

## 注意事项

- **代码审查不能替代测试** —— 两者是双轨；走完审查仍需走 104 测试工作流
- **审查是逆向阅读** —— 不假设代码对，假设它错，去找它哪里可能错
- **审查也要审"没写的代码"** —— 该有的边界处理、该有的日志、该有的测试有没有？
- **大 PR 是审查的最大敌人** —— 收到 >500 行变更的 PR，建议先拆分再审
