# 101 · 已有功能优化升级工作流程

> **触发条件**：开发者提出"优化、升级、改造、重构、提性能、提体验"等意图。
> **核心思想**：优化的最大风险是**破坏原有行为**。先理清现状，再设计变更，最后用回归证明无副作用。

---

## 步骤总览

```
(1) 优化目标确认  →  (2) 现状深度分析  →  (3) 优化方案设计  →  (4) 影响评估 + 用户确认  →
(5) 开发  →  (6) 三视角自审 + 回归  →  (7) 测试（进入 104）  →  (8) 审查（进入 103）  →  (9) 归档
```

---

## (1) 优化目标确认

### 做什么
明确本次属于以下哪一类，**不同类型的方案侧重不同**：

| 类型 | 目标 | 风险点 |
|---|---|---|
| 性能优化 | 接口/页面变快、资源占用变低 | 改写 SQL/缓存/异步逻辑可能引入数据一致性问题 |
| 体验优化 | 交互更顺畅、信息更清晰 | UI 改动可能影响已养成的用户习惯 |
| 功能增强 | 加字段、加分支、扩展场景 | 兼容老数据、兼容老接口调用方 |
| 技术重构 | 代码结构调整、不改外部行为 | 行为等价性难以证明，最易出回归 |

### 产出
- 文件：`docs/01_需求分析/yyyymmdd_xxx功能优化需求分析.md`
- 必须写清楚：
  - 优化前的**可量化指标**（接口耗时 X ms / 报错率 Y% / 体验痛点截图）
  - 优化后的**预期指标**
  - 业务/用户角度的成功标准

---

## (2) 现状深度分析（这一步是这个工作流的关键）

### 做什么
- 阅读现有代码，画出当前的：
  - 调用链 / 数据流
  - 数据模型与字段含义
  - 接口契约与所有调用方
  - 前端交互与状态
- 找出：
  - 已知技术债与隐患
  - 本次变更涉及的**数据关联与影响范围**
  - 已有功能的**完整用户使用场景**（不只是主路径）

### 产出
作为方案文档"现状分析"小节，必须包含：
- [ ] 调用链/数据流图（Mermaid）
- [ ] **本模块被谁调用** 的清单（grep 全仓代码搜接口路径、函数名）
- [ ] **本模块调用了谁**（依赖列表）
- [ ] 已有的相关测试用例清单
- [ ] 现有的兜底/降级/重试逻辑

### 红线
- 没做调用方清单 = 没做现状分析
- 看了一两个文件就开始写方案 = 不合格

---

## (3) 优化方案设计

### 做什么
- 阅读所有相关 `docs/00_通用规范/`
- 设计变更点，**逐项标注**：是新增 / 修改 / 删除？
- 标注**兼容性策略**：
  - 数据库字段改名 → 双写过渡？灰度迁移？
  - 接口入参变更 → 加版本号？保留旧字段并标 deprecated？
  - 前端组件 props 变更 → 全仓搜调用方？提供默认值？

### 产出
- 文件：`docs/02_功能设计/xx功能_优化方案_yyyymmdd.md`
- 模板：基于 `docs/20_模板/实施方案设计模板.md`，但需加 4 个特有小节：
  1. **优化目标与预期效果**（量化指标）
  2. **现状问题分析**
  3. **兼容性分析**（数据/接口/前端调用方分别说）
  4. **回归风险与应对策略**

---

## (4) 影响评估 + 用户确认 ⚠️ 卡点

### 做什么
向开发者**结构化呈现**：
- 改动了什么（清单）
- 影响了什么（调用方清单）
- 是否需要数据迁移
- 是否需要分批上线 / 灰度 / 开关
- 最坏情况下如何回滚

**得到明确确认后**才进入开发；更新 `docs/文档更新日志.md`。

---

## (5) 开发

### 做什么
- 按方案分步实现，**每完成一个子模块就跑一次相关测试**
- 兼容性代码（双写、双读、兼容旧字段）必须加 `// TODO: 兼容期到 yyyy-mm-dd 后删除` 这类**清晰的标记**
- 大重构按"小步快跑"：每一步可独立提交、可独立通过测试

### 红线
- 禁止在优化中**顺手扩展功能** —— 把功能扩展拆成独立任务，进入 100 工作流
- 禁止**清理无关代码**，本任务专注本任务

---

## (6) 三视角自审 + 回归

### 三视角
1. **错误视角**：本次代码自身有没有错？
2. **方案视角**：代码是否完全落实了方案？
3. **回归视角**（**本工作流的重点**）：
   - 受影响调用方是否都跑过一遍？
   - 现有测试用例是否全部通过？
   - 是否新增了"防止本次优化撞坏原功能"的测试？

### 工具
- 全仓 grep：被改动的函数名、SQL 表名、接口路径、组件名
- 跑历史回归用例：`docs/05_回归用例/` 与对应模块的实施方案末尾用例

---

## (7) (8) (9) 测试、审查、归档

- 测试：进入 [`104_测试工作流程.md`](104_测试工作流程.md)，**重点是回归测试，不是新功能测试**
- 审查：进入 [`103_代码审查工作流程.md`](103_代码审查工作流程.md)
- 归档：
  - [ ] 更新用户手册（如行为变化）
  - [ ] 优化方案文档补充"实施结果"小节，记录前后指标对比
  - [ ] 新增回归用例到 `docs/05_回归用例/`
  - [ ] `docs/文档更新日志.md`

---

## 注意事项

- **性能优化必须有 before/after 数据**，没有数据的"优化"不算数
- **重构不改外部行为是硬性要求**，行为变化属于功能升级，应拆出去
- **兼容期不能无限延长** —— 设置删除日期并跟踪
- **灰度/开关方案**：能用配置开关回滚的，永远比"代码层面回滚"更安全
- **DB 字段不要直接删** —— 先停止写入，留一段时间再删
