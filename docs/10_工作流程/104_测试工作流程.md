# 104 · 测试工作流程

> **触发条件**：开发者提出"加测试、跑回归、补用例"；或工作流 100/101/102 完成开发后强制进入。
> **核心思想**：**测试前置、风险驱动、可回放。** 测试不是"代码写完跑一遍"，是从需求阶段就介入的预防工作。

---

## 0. 触发与豁免

| 场景 | 是否必跑 |
|---|---|
| 新功能开发（100 流程） | 必跑，开发完成前完成测试设计 |
| 功能优化升级（101 流程） | 必跑，**重点是回归** |
| BUG 修复（102 流程） | 必跑，**必须新增回归用例** |
| 纯文档/注释变更 | 免跑，在提交说明中标注 `[no-test]` |
| 数据库结构变更 / 公共组件变更 | 必跑，加做迁移演练或全仓影响排查 |

> **AI 强制约束**：开发完成后必须主动产出本文档 §3 的测试用例与 §5 的自检清单，缺失即视为未完成。

---

## 1. 测试分层

按"测什么、谁主导、用什么手段"分层，**每一层目标不同，不能互相替代**：

| 层 | 目标 | 主导 | 入库位置 |
|---|---|---|---|
| L1 单元测试 | 单个函数/方法的纯逻辑分支 | AI + 开发自审 | 与源码同模块 |
| L2 组件/服务测试 | 服务协作、组件渲染交互 | AI + 开发 | 与源码同模块 |
| L3 接口契约测试 | REST 接口的入参/出参/错误码/权限 | AI 主导 | 后端 `api-test/` 或前端契约目录 |
| L4 集成测试 | 跨模块（消息队列 / 缓存 / 微服务）协作 | 开发 + 测试 | `docs/05_回归用例/` 关联 |
| L5 端到端 (E2E) | 用户视角的关键业务流 | 测试人员 | `docs/05_回归用例/E2E/` |
| L6 验收测试 | 产品/业务方确认功能达标 | 产品 | 需求文档附录 |

> **覆盖率底线建议**：核心业务逻辑 L1 行覆盖 ≥ 70%；每个接口至少 1 条 L3 happy path；金额/日期/权限相关方法 100% 覆盖边界。
> 覆盖率不是目标，**关键路径都被测过**才是目标。

---

## 2. 测试介入时机

```
需求评审 ──┐
           ├─▶ §3 可测性评审（卡点）
方案设计 ──┘
           ├─▶ §4 用例设计（卡点）
代码开发 ──┐
           ├─▶ §5 自测执行 + §6 自检清单（卡点）
代码自审 ──┘
           ├─▶ §7 回归 + §8 缺陷登记
PR/合并 ───┘
```

| 钩点 | 责任人 | 产出 | 卡点效果 |
|---|---|---|---|
| 需求评审完成 | AI / 测试 | 可测性意见 | 阻塞意见 → 退回产品 |
| 方案设计完成 | AI | 测试用例清单 | 用例缺失 → 不进入编码 |
| 代码开发完成 | AI | 单元/接口测试代码 + §6 自检 | 缺项 → 视为未完成 |
| PR 合并前 | 开发 + 测试 | 回归报告 + 缺陷登记 | 有未关闭 P0/P1 → 不合并 |

## 3. 可测性评审（需求/方案阶段）

收到需求或方案后，**动手前**先做体检：

- **验收用例是否能直接转写为测试步骤？** 模糊用语（"流畅、友好、合理"）必须换为可量化指标
- **边界条件是否明确？** 日期范围（开闭区间）、金额上限、字符串长度、列表为空、数据权限边界
- **异常路径是否覆盖？** 超时、并发冲突、外部依赖失败、权限不足、数据不存在
- **可观测性是否就绪？** 关键路径有日志/审计/埋点，失败时能定位
- **可回放性是否具备？** 是否需要种子数据；测试数据脱敏方案

输出位置：需求/方案文档末尾追加"测试视角评审记录"小节；**阻塞项需开发者答复后才能继续**。

## 4. 用例设计

### 4.1 用例来源（按优先级）

1. 需求文档的验收用例（100% 覆盖）
2. 业务流程图的每条主分支与异常分支
3. 数据权限矩阵的每一行
4. 历史 BUG 库（`docs/03_问题记录/`）的相似模块复发场景
5. 通用规范中的强制约束（特别是 `06_安全规范.md`、`07_错误处理与日志规范.md`）

### 4.2 设计方法

| 方法 | 适用 |
|---|---|
| 等价类划分 | 输入域大、规则简单（手机号、邮箱） |
| 边界值分析 | 数值/日期/分页（必做：min-1, min, min+1, max-1, max, max+1） |
| 判定表 | 多条件组合（权限 × 状态 × 角色） |
| 状态迁移 | 单据状态机（草稿→提交→审核→作废） |
| 错误推测 | 历史 BUG 模式（见 `docs/90_最佳实践/通用陷阱清单.md`） |

### 4.3 单条用例最小要素

- 用例编号（`TC-{模块}-{序号}`，如 `TC-USER-007`）
- 关联需求/BUG 编号
- 前置条件（数据、权限、登录态）
- 操作步骤（编号化）
- **预期结果**（具体断言点，**禁止"看起来对"**）
- 优先级（P0 阻塞 / P1 主流程 / P2 次要 / P3 体验）
- 自动化级别（手工 / 半自动 / 自动）

### 4.4 用例入库

- 新功能：用例清单写入实施方案文档末尾"测试用例"小节
- 优化：写入优化方案文档末尾
- BUG 修复：写入 BUG 记录文档"回归用例"小节，**同时拷贝到 `docs/05_回归用例/` 永久保留**

## 5. 自测执行

### 5.1 后端
1. 启动本地依赖（DB / 缓存 / MQ / 配置中心）
2. 跑单元测试 → L1/L2 全绿
3. 跑接口测试 → L3 全绿
4. 若涉及数据库迁移：在与生产同版本的空库执行迁移，再用历史数据快照库再跑一次
5. 关键查询开慢日志，确认无 N+1、无全表扫
6. 关键写入路径：用事务测试模拟回滚，验证一致性

### 5.2 前端
1. 单元测试 / 类型检查 / Lint 通过
2. 构建（build）通过且无 type error
3. **手动**在 dev server 走主流程一遍
4. 刻意触发：网络断开、慢网络（DevTools throttle）、空数据、超长数据、权限不足登录态
5. 检查浏览器 Console：**禁止**红色错误、未捕获 Promise rejection；Network 面板**禁止** 4xx/5xx（业务正常路径下）
6. 表单页：刻意提交空表单 + 非法值 + 提交后立即再点（防重）

### 5.3 跨端联调
- 接口字段命名、类型、可空性以 API 规范为准；前后端用 OpenAPI / 类型定义对照
- 日期统一时区、统一格式（ISO-8601）；金额统一精度

## 6. 自检清单（交付前硬性卡点）

> 任何"否"必须先解决再交付。

### 6.1 通用
- [ ] 所有验收用例都有对应测试用例并已通过
- [ ] 主流程 + 至少一条异常分支已自动化（单元或接口层）
- [ ] 无新增编译/类型/Lint 警告（或有明确豁免理由）
- [ ] 改动涉及的相似代码已 grep 搜索（防止"修一处漏十处"）
- [ ] 用户手册已同步更新

### 6.2 数据与边界
- [ ] **日期范围**：开闭区间在 SQL / 后端 / 前端三端一致
- [ ] **小数 ↔ 百分比**：转换只发生一次，无重复 × 100 / ÷ 100
- [ ] **金额**：后端用高精度类型，序列化为字符串避免精度丢失；前端千分位、等宽字体、右对齐
- [ ] **日期格式**：展示与传输格式明确（建议 ISO-8601 传输）；时区在边界层统一处理
- [ ] **分页**：page 起始（0 还是 1）前后端一致；`total/hasNext` 有断言
- [ ] **排序**：默认排序字段稳定（带 ID 作为 tiebreaker），翻页不串行
- [ ] **空值**：null / "" / 0 / [] 行为有明确预期，UI 不出现 "undefined" / "null" 字样
- [ ] **超长**：文本输入达字段上限不报 500；列表显示有截断与 hover 全文
- [ ] **大数据量**：列表/导出 1w+ 行不卡死（虚拟滚动 / 分批 / 流式）

### 6.3 权限与安全
- [ ] **功能权限**：每个按钮/菜单在权限矩阵中有明确角色
- [ ] **数据权限**：行级过滤在 SQL 层（不是 Service 内存过滤），并有测试
- [ ] **字段脱敏**：脱敏在序列化层，而非依赖前端隐藏
- [ ] **越权**：构造非所有者角色的请求，断言 403 / 数据不可见
- [ ] **输入校验**：后端有二次校验，禁止仅依赖前端校验
- [ ] **日志**：禁止打印明文密码、token、身份证、手机号

### 6.4 并发与一致性
- [ ] **写入**：有乐观锁或唯一索引，模拟并发提交断言一次成功一次失败
- [ ] **防重**：提交按钮 loading + 后端幂等
- [ ] **事务**：跨表写入在同一事务内；外部调用不卷入数据库事务
- [ ] **消息**：消费者幂等

### 6.5 前端交互
- [ ] **三态齐全**：加载态、空态、错误态都有明确 UI
- [ ] **响应式**：常见分辨率（PC / 笔记本 / 移动）下不破版
- [ ] **可访问性**：键盘导航、焦点可见、关键控件有 aria 标签
- [ ] **导出**与列表当前筛选条件一致

## 7. 回归测试

### 7.1 何时做
- L2 及以上层级代码改动
- 公共组件 / 工具函数 / SQL 公共片段 / 拦截器 / 全局配置变更
- 数据库迁移新增

### 7.2 怎么做
1. **影响域分析**：grep 搜符号/SQL 表名/API 路径，列出受影响模块
2. **从历史用例库挑选**：`docs/05_回归用例/` 与相关模块的实施方案末尾用例
3. **跑自动化**：先跑受影响模块的 L1/L2/L3，再跑相邻模块 L3
4. **关键流程人工烟测**：登录 → 主菜单 → 受影响模块 → 关联报表/导出
5. **回归结论写入 PR 描述**

## 8. 缺陷登记

| 严重级别 | 定义 | 处置 |
|---|---|---|
| P0 阻塞 | 功能完全不可用 / 数据错乱 / 安全漏洞 | 立即停下当前开发，优先修复 |
| P1 主流程 | 主流程缺陷有规避方法 | 当前迭代必须修复 |
| P2 次要 | 边缘场景、提示不准确 | 排入下一迭代 |
| P3 体验 | UI 偏差、文案问题 | 累积处理 |

- 新缺陷统一记录到 `docs/03_问题记录/yyyymmdd_BUG_xxx.md`
- 修复后**必须新增一条回归用例**到 `docs/05_回归用例/`
- 同一根因影响多处：**在同一文档内**列出所有受影响位置，避免拆分多个 BUG 单失去关联

## 9. AI 生成测试的强制约定

AI 自动生成测试代码时必须满足：

1. **命名清晰**：`should_<行为>_when_<前置条件>` 或 `测试_当_X_应该_Y` 这类自解释格式
2. **独立可乱序**：每个测试自带数据准备与清理
3. **明确断言**：每个测试至少一个明确断言；**禁止**仅断言"没抛异常"
4. **真实数据源**：禁止依赖生产数据 dump；使用工厂方法/Builder 生成测试数据
5. **Given/When/Then 结构**：注释或空行分隔，三秒看懂用例意图
6. **真实环境**：涉及 DB 的测试用真实数据库（容器化），**禁止**用方言不同的内存 DB 模拟
7. **不可哑测**：禁止只为凑覆盖率、断言为 `assertNotNull(result)` 的"空壳测试"

## 11. 测试报告最小模板（贴入 PR）

```markdown
## Test plan

- [ ] 单元测试：xxx 模块新增/修改 N 条，覆盖率 NN%
- [ ] 接口测试：POST /api/v1/xxx happy + 4xx + 权限场景
- [ ] 手工烟测：登录 → 进入 X 页面 → 执行 Y → 校验 Z
- [ ] 回归：相邻模块 A、B 主流程跑通
- [ ] 数据迁移：在 vX.Y 快照库执行成功，无锁等待 > 1s
- [ ] 自检清单（§6）逐条通过
```

---

## 注意事项

- **测试不是负担，是保险** —— 写测试的时间一定比线上救火的时间短
- **"已测过"不等于"已通过"** —— 失败的测试也要在 PR 中说明结论
- **不要被覆盖率绑架** —— 高覆盖率的烂测试不如低覆盖率的好测试
- **测试代码也要审** —— 测试代码是项目的一等公民，质量同样重要
