# 100 · 新功能开发工作流程

> **触发条件**：开发者提出"新增功能、从零做一个、加一个 xx 模块"等意图。
> **核心思想**：先想清楚、再写代码、边写边审、最后归档。**禁止收到指令直接开写。**

---

## 步骤总览

```
(1) 需求分析  →  (2) 方案设计  →  (3) 方案审查 + 用户确认  →
(4) 开发  →  (5) 代码自审  →  (6) 测试（进入 104）  →  (7) 代码审查（进入 103）  →
(8) 用户手册 / 归档
```

---

## (1) 需求分析

### 做什么
- 阅读开发者提供的需求描述、UI 图、PRD（若有）
- 提取：业务目标、用户角色、核心场景、约束条件、验收标准
- **凡是有歧义或缺失之处，列成问题清单回问开发者，禁止猜测**

### 产出
- 文件：`docs/01_需求分析/yyyymmdd_xxx需求分析.md`
- 模板：`docs/20_模板/需求分析模板.md`

### 自检
- [ ] 用户角色 / 权限点已枚举
- [ ] 核心业务流程已用流程图（Mermaid）画出
- [ ] 边界与异常路径已列出（非空、超长、并发、超时、权限不足）
- [ ] 验收标准是**可执行的测试步骤**，不是"流畅、友好"这类模糊词
- [ ] 与开发者澄清的问题已记录在文档"待确认事项"小节

---

## (2) 方案设计

### 做什么
- 阅读 `docs/00_通用规范/` 中所有相关规范（数据库、API、前端、后端、安全）
- 阅读现有代码，找到本次新功能要嵌入的位置和依赖
- 设计：数据模型 → 接口契约 → 后端实现路径 → 前端页面结构 → 关键算法 / 状态机
- 评估影响范围：哪些已有模块会被改动？是否有兼容性风险？

### 产出
- 文件：`docs/02_功能设计/xx功能_实施方案_yyyymmdd.md`
- 模板：`docs/20_模板/实施方案设计模板.md`

### 自检
- [ ] 数据库变更（建表/加字段/索引）写清楚 DDL，**含回滚策略**
- [ ] 每个新接口写清：路径、方法、入参、出参、错误码、权限点
- [ ] 前端涉及的页面、组件、路由、状态、交互都画出来
- [ ] 列出"本方案不做什么"，避免范围蔓延
- [ ] 风险与降级方案各列至少 1 条
- [ ] 末尾追加"测试用例清单"（详见 104 工作流 §4）

---

## (3) 方案审查 + 用户确认 ⚠️ 卡点

### 做什么
- 把方案以**结构化摘要**形式发给开发者（不要直接丢一大段文档让他自己看）
- 主动提出本方案的争议点、不确定项、影响范围
- **等待开发者明确说"确认 / 开始 / 可以 / 同意"** 之类的字样

### 卡点规则
- ❌ 开发者只说"嗯"、"好的"、"知道了"——**不算确认**，需要明确表态
- ❌ 影响范围大但方案被秒批——主动追问"是否注意到 X 影响？"
- ✓ 收到明确确认后，先更新 `docs/文档更新日志.md`，再进入开发

---

## (4) 开发

### 做什么
- **每动一个领域的代码前，再读一遍对应规范**（命名、API、DB、前端、后端、安全）
- 按规范实现，包括：命名、目录结构、错误处理、日志、注释（**不写废话注释**）
- 边写边运行/编译，**禁止整批代码写完才第一次跑**
- 使用版本控制，按 `docs/00_通用规范/08_Git协作规范.md` 提交

### 红线
- 禁止在本任务范围外"顺手优化"
- 禁止跳过规范"先跑通再说"——跑通后不会回来补
- 禁止用 `console.log` / `print` 做日志（除非项目规范明确允许）
- 禁止吞掉异常（`catch` 后必须 log + 抛出 / 处理）

---

## (5) 代码自审（开发完毕立即做）

### 三视角自审
1. **代码视角**：是否有错误？类型、空指针、死循环、资源泄漏、N+1 查询？
2. **业务视角**：是否实现了方案的全部内容？验收标准能否一条条勾掉？
3. **回归视角**：本次改动是否触碰了公共代码？相同模式的代码是否需要一并修改？（**grep 搜一遍**）

### 自检
- [ ] 编译/类型检查 0 错误 0 警告（或警告有明确豁免理由）
- [ ] Lint / Format 通过
- [ ] 主流程在本地手工跑了一遍
- [ ] 至少一条异常路径手工跑了一遍
- [ ] 浏览器/服务端日志无未处理的红色错误
- [ ] 改动文件列表与方案匹配，无范围外文件

## (8) 归档

### 必做
- [ ] 用户手册：`docs/10_用户手册/yyyymmdd_xxx功能说明.md`（新建）
- [ ] 实施方案文档"测试用例"小节补齐执行结果
- [ ] `docs/文档更新日志.md` 追加本次变更条目

### 按需做
- [ ] 若涉及字典新增/变更 → 更新 `docs/11_字典/系统字典.md`
- [ ] 若涉及菜单、角色、权限、数据权限配置 → 在用户手册末尾的"管理员配置"小节写清楚
- [ ] 若涉及对外 API → 同步更新 OpenAPI / 接口文档

## 注意事项

- **可观测性别忘记**：关键路径要有日志或埋点，否则上线后出问题无法定位
- **国际化别忘记**：项目支持多语言时，新增文案必须走 i18n
- **空态/加载态/错误态**：前端三态齐全是基本要求，不是"额外功能"
- **数据权限别只做功能权限**：列表过滤了，导出/详情接口也要过滤
- **写日志/写注释/写文档**：是开发的一部分，不是"有时间再补"
