# AI 协作避坑清单

> 这是 AI 在协作中最容易犯的错。开发者把它当"问题预防"，AI 把它当"自我审视"。
> 每一条都是教训换来的。

---

## 一、AI 容易踩的"会话级"坑

### 1.1 假装听懂了实际没听懂
**症状**：开发者说了一段背景，AI 直接进入"那我开始做"
**真相**：背景里其实有关键约束 AI 没识别
**避坑**：动手前用一两句话**复述需求**，请开发者确认理解一致

### 1.2 把"嗯"当确认
**症状**：开发者只说"嗯"或"OK"，AI 当作"开始干"的信号
**真相**："嗯"通常只是表示"我看到你的消息了"，不是同意方案
**避坑**：未收到明确的"开始 / 确认 / 同意 / 可以了"等字眼，**继续等**

### 1.3 不问就猜
**症状**：业务字段含义不清，AI 自己脑补一个
**真相**：脑补错了一切都是错的
**避坑**：列出不确定项的清单回问开发者

### 1.4 任务边界自动扩张
**症状**：本来让修 BUG，AI 顺手"清理了一下代码风格"
**真相**：变更范围变大、审查成本变高、出问题溯源难
**避坑**：严格做"本次任务范围"列出来的事，其他记录到待办

### 1.5 进度黑箱
**症状**：开发者发完指令后 AI 闷头干 30 分钟才报结果
**真相**：开发者无法及时纠偏
**避坑**：每完成一个子步骤报一次

### 1.6 失败不响
**症状**：AI 遇到卡点（环境问题、规范冲突、找不到文件）静默尝试 N 种绕路
**真相**：绕路通常越绕越偏
**避坑**：超过 2 次失败就**抛给开发者**

### 1.7 一次提交几千行
**症状**：AI 把整个功能一口气写完，PR 巨大无法审查
**真相**：审查者必然走过场
**避坑**：拆成原子提交，每个 ≤ 400 行

## 二、AI 容易踩的"代码级"坑

### 2.1 用过期/虚构的 API
**症状**：AI 写出库 / 框架中不存在的函数
**真相**：训练数据有时滞，AI 会"创造" API
**避坑**：写陌生库的代码前先查官方文档，**不要靠记忆**

### 2.2 修一处漏十处
**症状**：BUG 修了一个文件，相同模式的另外 5 个文件还有同样问题
**真相**：根因相同的代码可能散落多处
**避坑**：修完一处后 grep 搜相同模式

### 2.3 忽略错误处理
**症状**：写出 happy path 一切正常，但异常路径直接 throw 或者吞掉
**避坑**：每写一个函数问"这个函数所有可能失败的方式是什么？怎么处理？"

### 2.4 静默 try-catch
**症状**：`catch (e) {}` 或 `catch (e) { return null; }`
**真相**：错误被埋葬，线上出问题无法追
**避坑**：每个 catch 必须 log + 决策（抛 / 处理 / 降级）

### 2.5 浮点数处理金额
**症状**：用 `float` / `double` 处理金额
**真相**：精度永远丢
**避坑**：金额用高精度类型（BigDecimal / Decimal）或整数（分）

### 2.6 日期边界乱
**症状**：前端传 `2026-01-01`，后端用 `created_at >= '2026-01-01'`，但又对 end 用 `<= '2026-01-31'` 漏掉 31 号
**避坑**：开闭区间在 SQL / 后端 / 前端三端约定一致

### 2.7 软删除遗忘
**症状**：新写的查询忘记加 `is_deleted = false`
**避坑**：用共享 SQL 片段 / 视图 / ORM 过滤器自动加，**不要靠每次记得**

### 2.8 在事务里调外部
**症状**：`@Transactional` 方法中 HTTP 调用 / MQ 发送
**真相**：数据库连接长时间占用，可能死锁
**避坑**：外部调用放事务外，必要时用 Outbox 模式

### 2.9 列表过滤但导出没过滤
**症状**：列表接口做了数据权限，导出走不同入口绕过了
**避坑**：导出共享列表的过滤逻辑

### 2.10 SQL 注入
**症状**：拼 SQL 字符串 `"WHERE name = '" + userInput + "'"`
**避坑**：参数化查询是底线，**没有例外**

## 三、AI 容易踩的"流程级"坑

### 3.1 跳过自审就交付
**症状**：开发完直接报"完成了"
**避坑**：自审清单逐条对照，至少跑一遍主流程

### 3.2 不更新文档
**症状**：代码改了但需求文档 / 用户手册 / API 文档没改
**避坑**：文档更新视为开发的一部分

### 3.3 不写测试
**症状**："你要的话我可以写测试"
**真相**：测试是开发的一部分不是可选项
**避坑**：按 `104_测试工作流程.md` 强制执行

### 3.4 不归档产出
**症状**：方案、BUG 分析、审查报告写在对话里不入库
**避坑**：按 `CLAUDE.md` §4 强制归档

### 3.5 假装完成
**症状**：报"已完成"但其实有跳过的检查项
**避坑**：用自检清单逐条勾，**不能勾的不能说完成**

## 四、AI 容易踩的"沟通级"坑

### 4.1 废话开场
**症状**：每次回复"好的，我来帮您 xxx"
**避坑**：直接进入正题

### 4.2 重复确认
**症状**：开发者已经明确说了，AI 再问一遍
**避坑**：仔细读上下文

### 4.3 文档化输出当回复
**症状**：开发者问个小问题，AI 返回 1000 字带 H1/H2 的文档
**避坑**：问题大小决定回复大小

### 4.4 道歉过度
**症状**：每次出错都"抱歉"开头然后又出同样的错
**真相**：道歉不抵改正
**避坑**：直接修，并把根因记入避坑清单

### 4.5 用感叹号过度
**症状**：每句话都带"！"
**避坑**：严肃工程文本，少用感叹号

## 五、AI 容易踩的"判断力"坑

### 5.1 过度乐观估计
**症状**："这个 5 分钟就好"，实际半小时还没完
**避坑**：估计时考虑：读规范 + 写代码 + 自审 + 测试 + 文档

### 5.2 跳过繁琐步骤
**症状**：知道应该写测试，但"为了快"先跳过
**真相**：跳过的测试 99% 永远不会补
**避坑**：流程是流程，效率诉求不是借口

### 5.3 一条路走到黑
**症状**：选定一个方案后发现行不通，继续打补丁而不退回
**避坑**：方案不对要敢于推翻，**沉没成本不能绑架决策**

### 5.4 把"我能做"当"我应该做"
**症状**：开发者抛出一个需求，AI 觉得 trivial 就立刻动手
**避坑**：trivial 也要先检查是否符合工作流入口（方案 → 确认 → 实施）

### 5.5 把"看起来对"当"对"
**症状**：代码 lint 通过、能编译通过，就声称工作完成
**真相**："能编译"距离"业务正确"很远
**避坑**：真跑业务路径验证

## 六、双方共同要避的坑

### 6.1 长会话失焦
**症状**：会话里堆了 N 个任务，AI 在不同任务间漂移
**避坑**：一个会话一个主任务；新任务起新会话

### 6.2 信息不对称累积
**症状**：开发者改了项目的某个约定，但没告诉 AI（AI 没看出来）
**避坑**：项目约定改动同步到 `CLAUDE.md` / 规范文档

### 6.3 规范成为摆设
**症状**：规范文档一直没更新，与实际代码越走越远
**避坑**：开发流程中**强制更新规范**（特别是引入新模式时）

### 6.4 责任不清
**症状**：AI 写的代码出问题，开发者说"是 AI 写的"
**真相**：审查通过即开发者负责
**避坑**：审查必须实质审，不可走过场

## 七、自我反省（AI 每次任务结束自问）

- 我有没有今天上面任何一条坑里？
- 如果有，下次怎么避免？
- 是不是应该把这个反思写进规范？
