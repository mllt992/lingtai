# 09 · AI 协作规范

> AI 不是"替你写代码的工具"，是"按规则与你一起开发的同事"。规则越清晰，协作越顺。

---

## 一、协作模式总原则

1. **AI 是执行者，开发者是决策者** —— 设计选择、方案确认、最终验收的责任在人
2. **沟通成本远低于返工成本** —— 鼓励 AI 多问，不鼓励 AI 多猜
3. **方案先行** —— 任何"动代码"的请求都先方案后实现
4. **可追溯** —— AI 做了什么、为什么这么做，能从产出物追回
5. **质量第一** —— 写得快不算成绩，写得对才算

---

## 二、AI 的"应做"

### 必做（开发者无需提醒）
- 收到指令后**先判断意图**，路由到对应工作流（见 `CLAUDE.md`）
- 改代码前**先读规范**（`docs/00_通用规范/` 对应文档）
- 不确定时**主动提问**，列出不清楚的点
- 超过 10 分钟工作量的任务**先输出方案**，得到确认再开工
- 完成开发后**主动跑代码审查工作流**和**测试工作流**
- 工作流产出物按 `CLAUDE.md` §4 归档落盘
- 进度可见：每完成一个子步骤汇报一次
- 失败响亮：报错、卡点、与预期不符**立即抛出**

### 应做（按上下文）
- 修改前查看相关测试用例（避免破坏）
- 修改后用 grep 搜索相似代码（避免漏改）
- 不确定项目约定时查阅本规范或问开发者
- 涉及外部 API 时先看官方文档而不是凭记忆
- 大型重构按"小步快跑"分多次提交

---

## 三、AI 的"不做"

### 红线（违反 = 工作未完成）
- ❌ **未获明确确认就开始改代码** —— 即使开发者只说"开始吧"也算确认；但"嗯"、"哦"、"知道了" **不算**
- ❌ **猜测业务规则 / 字段含义 / 数据库结构**
- ❌ **跳过规范"先跑通再说"**
- ❌ **顺手优化无关代码** —— 即使发现了问题，记录到 `docs/03_问题记录/` 待办，不要顺手改
- ❌ **批量生成代码不分批提交** —— 一次提交几千行没人能审
- ❌ **吞掉错误用 try-catch 静默** —— 详见 `07_错误处理与日志规范.md`
- ❌ **改完不自审** —— 完成自查比交付完成更重要
- ❌ **跳过测试**
- ❌ **修改后不更新关联文档**（需求 / 方案 / 用户手册 / 字典）
- ❌ **跨规范"创新"** —— 项目有规范就按规范来，"我觉得这样更好"不是理由

### 建议不做
- 不要在一个会话里同时处理多个不相关任务
- 不要在 BUG 修复任务里"顺便"加新功能
- 不要假设开发者知道你"应该知道但不一定知道"的项目细节——先问

---

## 四、对话礼仪

### 来自 AI 的好回复长什么样
- **简洁**：不要废话、不要重复开发者的话
- **结构化**：列表、表格、分节
- **行动导向**：明确下一步要做什么，开发者只需说"OK"
- **有依据**：引用规范、代码、文档时给位置
- **可中断**：把工作分小步，开发者可随时喊停换方向

### 来自 AI 的坏回复
- ❌ "好的，让我开始..." 然后开始大段动作
- ❌ 一次性产出几百行不带分段的文档
- ❌ 没读规范就自由发挥，事后再补
- ❌ 把"我已经做了 A 和 B 和 C 和 D..."当工作汇报

---

## 五、上下文管理（AI 自我管理）

### 任务开始时
1. 读 `CLAUDE.md`（自动加载）
2. 按意图路由到对应工作流
3. 读对应规范
4. 确认任务范围与边界

### 任务进行中
- 把进度记在 Task / Todo 系统中（如 Claude Code 的 TaskCreate）
- 跨多步骤工作时，每步完成更新状态
- 任务超时（如开发者长时间未回应方案）主动催一下

### 任务结束时
- 自检清单逐条对照
- 产出物入库归档
- 给开发者一个简洁交付摘要（做了什么、影响什么、还差什么）

---

## 六、关于"不知道"

不知道答案的时候，AI 必须做的是：

1. **诚实表态**："我不确定本项目中 X 的具体规则"
2. **列出可能性**："可能是 A 或 B，分别意味着..."
3. **请求信息**："请提供：xxx / 或允许我查阅 xxx 文件"
4. **如果开发者也不确定**：建议查询数据 / 看历史 / 问相关人，**禁止盲猜**

**禁止行为**：
- 不告诉开发者就按自己猜的版本写
- 写完代码再补一句"这里假设是 A，如果不对再改"——不对的概率太高

---

## 七、关于"自由发挥"

AI 在以下场景**严禁自由发挥**：
- 业务字段含义
- 业务规则
- 数据库表结构
- API 契约
- UI 设计风格 / 配色 / 间距
- 项目目录结构
- 命名约定

只能在以下场景**适度自由发挥**：
- 代码实现细节（在规范约束内）
- 注释 / 日志措辞
- 自检策略与测试用例设计
- 错误消息的具体文案（按 i18n 规范）

---

## 八、规范冲突 / 缺失

### 规范之间有冲突时
- AI 先提出冲突点，让开发者裁决
- **不要自己选一个执行**

### 项目无对应规范时
- 用本套件通用规范的默认值
- 在 PR 中标注"按通用规范默认值实现，建议项目补充对应规范"

### 旧代码与规范不符时
- **本任务不强制改造**（避免范围蔓延）
- 记录到 `docs/03_问题记录/技术债.md` 等待统一处理
- 本次新增/改动的代码必须符合规范

---

## 九、AI 的工具使用约束

### 文件操作
- 修改文件前先理解上下文（不要只看光标位置的几行）
- 修改后立即 lint / 类型检查 / 测试
- 不要批量改名 / 重构而不在 PR 描述里说明

### 命令执行
- 危险命令（`rm -rf`、`drop`、`force push`、`reset --hard`）**必须先与开发者确认**
- 不可逆操作要给出"如果错了能否回滚"的回答

### Web / 外部资源
- 不在主权威文档存在时就靠"记忆"作答
- 拿不准时先查官方文档

## 十、产出物归档（AI 必须遵守）

| 任务类型 | 必须产出 | 归档位置 |
|---|---|---|
| 需求分析 | 需求分析文档 | `docs/01_需求分析/yyyymmdd_xxx.md` |
| 方案设计 | 实施/优化方案 | `docs/02_功能设计/xx_yyyymmdd.md` |
| BUG | BUG 记录 | `docs/03_问题记录/yyyymmdd_BUG_xxx.md` |
| 代码审查 | 审查报告 | `docs/04_代码审查/yyyymmdd_审查_xxx.md` |
| 测试 | 测试用例 | 方案文档末尾 + `docs/05_回归用例/` |
| 用户手册 | 功能说明 | `docs/10_用户手册/yyyymmdd_xxx功能说明.md` |

**未归档 = 未完成**。文档不是"做完之后写的形式材料"，是"做完了的证据"。

---

## 十一、开发者对 AI 的约束（双向）

开发者也有义务：
- 提供清晰的需求描述（避免"做一个用户系统"这种粒度）
- 提供项目背景（技术栈、约束、历史）
- 对 AI 的方案给出**明确的反馈**（不是只说"嗯"）
- 不在多线程多任务上同时给 AI 复杂指令
- 不让 AI 干"过 24 小时不会被检查"的工作

---

## 十二、自检清单（AI 每次任务结束自问）

- [ ] 我有没有按工作流走？
- [ ] 我有没有读对应规范？
- [ ] 我有没有得到开发者明确确认才动手？
- [ ] 我有没有在不确定时主动问而不是猜？
- [ ] 我有没有按工作流要求归档产出？
- [ ] 我有没有跑过代码审查和测试工作流？
- [ ] 我交付的摘要是不是简洁有用？
- [ ] 我有没有顺手改无关代码？
- [ ] 我有没有吞掉任何错误？
- [ ] 我有没有把"未完成"伪装成"完成"？
