# 07 · 错误处理与日志规范

> 系统能不能用，不取决于没出错的那 99%，取决于出错时的那 1% 能不能定位、能不能恢复。

---

## 一、错误处理总原则

1. **快速失败（Fail Fast）** —— 早发现早暴露，不要让错误传播得很远才爆
2. **不吞异常** —— `catch` 永远要么 log + 抛出、要么显式处理（带注释说明为什么吞）
3. **错误信息有用** —— "出错了"等于没说，要带：发生了什么、上下文、可追踪 ID
4. **可恢复 vs 不可恢复区分** —— 业务异常面向用户，系统异常面向运维

---

## 二、异常分类

### 业务异常（Business Exception）
- 用户操作不合理导致：参数错误、状态不允许、权限不足
- 应该返回 4xx
- 包含明确的业务错误码、面向用户的可读消息
- **不打 ERROR 日志**（这是预期内的错误），用 INFO/WARN

### 系统异常（System Exception）
- 编程错误、外部依赖故障、未预期情况
- 应该返回 5xx
- 必须打 ERROR 日志，含完整堆栈和上下文
- 前端用户看到的是统一友好提示 + traceId，**不暴露堆栈**

### 临时异常（Transient Exception）
- 网络抖动、超时、限流、服务暂时不可用
- 可重试（仅幂等操作）
- 重试失败转为系统异常

## 三、统一错误响应

### 结构
```json
{
  "code": "USER_NOT_FOUND",
  "message": "用户不存在",
  "details": [
    { "field": "email", "reason": "格式不合法" }
  ],
  "traceId": "abc123-xxx",
  "timestamp": "2026-05-13T10:00:00Z"
}
```

### 字段说明
- `code`：可枚举的错误码字符串，全项目统一维护
- `message`：用户可读消息（注意 i18n）
- `details`：字段级错误（可选）
- `traceId`：贯穿请求的追踪 ID
- `timestamp`：发生时间

### 错误码规范
- 命名：`<模块>_<错误描述>` 或全大写下划线：`USER_NOT_FOUND`、`ORDER_STATUS_INVALID`
- 维护：错误码字典 `docs/11_字典/错误码字典.md`
- 数字 code 也可（如 `1001`），但字符串可读性更好

### 红线
- ❌ 业务失败返 200 + body 里 `success: false` （历史遗留可保留，新项目禁止）
- ❌ 把堆栈/SQL 错误直接返回前端
- ❌ 错误消息暴露内部实现（"NullPointerException at xxx.java:42"）

## 四、异常捕获模式

### 模式 A：让它抛
- 框架/语言已经有合理处理（如 Spring 的全局异常处理器、Express 的 errorHandler）
- 让异常自然抛出到统一处理器
- **不要每个方法都加无意义的 try-catch**

### 模式 B：转译
- 底层异常需要包装为业务异常时：
  ```
  catch DataAccessException → throw BusinessException("DATA_ACCESS_FAILED")
  ```
- 包装时**保留原始异常**作为 cause，避免堆栈断链

### 模式 C：兜底
- 业务可降级的场景：
  ```
  try { useCache() }
  catch (e) {
    log.warn("cache failed, fallback to db", e);
    useDb();
  }
  ```
- **必须 log**，否则降级悄无声息无法发现

### 模式 D：明确吞
- 真的不需要处理的场景（极少）：
  ```
  try { closeResource() }
  catch (e) {
    // 关闭失败不影响主流程，但要记一条日志便于排查泄漏
    log.warn("resource close failed", e);
  }
  ```
- **必须有注释说明为什么可以吞**

### 红线
- ❌ 空 catch：`catch (e) {}`
- ❌ catch 后 swallow + 返回默认值（业务变成静默失败）
- ❌ catch 后只 print 不 throw 不 log（错误消失）
- ❌ catch Throwable / Exception 然后什么都不做

## 五、日志规范

### 等级
| 等级 | 用途 | 何时使用 |
|---|---|---|
| TRACE | 极细粒度调试 | 几乎不用 |
| DEBUG | 开发调试 | 排查时临时开，生产关闭或采样 |
| INFO | 业务关键事件 | 订单创建、状态变更、定时任务开始/结束 |
| WARN | 异常但可恢复 | 重试成功、降级生效、限流命中、参数异常但有默认值 |
| ERROR | 未预期错误 | 异常导致功能失败，必须人工关注 |
| FATAL | 致命错误 | 进程要挂、数据要丢，极少用 |

### 必含字段
所有日志（特别是 INFO+）必含：
- 时间戳
- 等级
- traceId / requestId（贯穿一次请求）
- 业务实体 ID（订单号、单据号、用户 ID，如有）
- 模块/类名

### 结构化日志
- 强烈建议结构化日志（JSON），便于 ELK / Loki 查询
- key 命名一致：`order_id`、`user_id`、`amount`

### 日志内容
- INFO：写发生了什么（**一句话能说清**），关键参数（脱敏）
  ```
  INFO order created order_id=12345 user_id=789 amount=199.00
  ```
- WARN：写"异常但已处理"
  ```
  WARN retry succeeded after 2 attempts api=user-service endpoint=/profile
  ```
- ERROR：写"什么失败、为什么、堆栈、上下文"
  ```
  ERROR payment failed order_id=12345 reason="gateway timeout" {stack}
  ```

### 红线
- ❌ 打印明文密码、token、密钥、身份证、银行卡（详见 `06_安全规范.md` §七）
- ❌ ERROR 不带堆栈
- ❌ 在循环里打 INFO（一次请求几千条日志）
- ❌ 用 `print` / `console.log` 代替日志框架
- ❌ 一个错误打两次（catch 处 log 一次，外层处理器又 log 一次）

## 六、追踪（Tracing）

### TraceId 贯穿
- HTTP 请求入口生成 traceId，写入 MDC / context
- 所有日志携带 traceId
- 跨服务调用通过 HTTP Header（`X-Trace-Id`、`traceparent`）传递
- 异步任务、消息消费要从消息中恢复 traceId

### 分布式追踪
- 大项目接入 OpenTelemetry / Jaeger / Zipkin / Datadog APM
- 关键调用点埋点（DB、Cache、RPC、MQ）

---

## 七、监控与告警

### 指标
- 接口 QPS / 错误率 / P95 / P99
- DB 慢查询数量
- 缓存命中率
- 队列长度 / 消费延迟
- JVM/Node/Python 进程指标（CPU、内存、GC）

### 告警
- ERROR 日志陡增告警
- 错误率超阈值告警
- P95 耗时超阈值告警
- 队列堆积告警
- **告警要分级**（P0 立刻响应 / P1 工作时间响应 / P2 看板观察）
- 避免告警风暴：合并、抑制、去重

---

## 八、前端错误处理

### 全局错误捕获
- `window.onerror` / `unhandledrejection` 上报
- Vue / React error boundary 防整页崩溃
- Sentry / 自建 SDK 上报

### 错误展示
- 友好提示（不是技术错误信息）
- 提供后续动作（重试、返回、联系客服）
- 重要错误展示 traceId 便于反馈

### 红线
- ❌ 把 5xx 响应的原始内容 alert 给用户
- ❌ 报错就空白页（用 error boundary 兜底）

---

## 九、Panic / 进程崩溃恢复

- 进程级未捕获异常必须有兜底（不能让进程因一个请求挂掉）
- 容器编排（K8s）开启健康检查 + 自动重启
- 关键服务有多副本，单实例挂掉不影响整体可用性

---

## 十、可测试性

- 自定义异常类带等价比较，便于在测试里断言
- 日志框架支持测试时捕获（断言 "应该打了 ERROR"）

---

## 十一、自检清单

### 错误处理
- [ ] 业务异常 → 4xx + 业务码 + 友好消息
- [ ] 系统异常 → 5xx + traceId + 不暴露堆栈给前端
- [ ] 全局异常处理器已配置
- [ ] 错误码字典已更新
- [ ] 无空 catch、无吞掉错误

### 日志
- [ ] 关键业务事件有 INFO 日志（订单创建、状态流转、登录登出等）
- [ ] 异常路径有 WARN / ERROR 日志
- [ ] ERROR 日志带完整堆栈和业务上下文
- [ ] 日志带 traceId、userId、业务 ID
- [ ] 日志无敏感明文
- [ ] 日志输出到中央日志系统（ELK / Loki / 云日志）

### 追踪与监控
- [ ] HTTP 请求入口生成 traceId
- [ ] traceId 跨服务、跨异步传递
- [ ] 关键接口有耗时和错误率监控
- [ ] 告警配置就绪
