# 01 · 命名与代码风格规范

> 命名是工程中最便宜的可读性投资，也是最容易省错的部分。

---

## 一、命名总原则

1. **见名知意** —— 读到名字能猜到含义，无需打开文件追溯
2. **避免缩写** —— 除业内通用缩写（id、url、http、api、db、ui、ip），其余一律全拼
3. **一致优于聪明** —— 同项目同概念用同一个词；项目里叫 `customer` 就别再叫 `client` 或 `user`
4. **领域语言优先** —— 业务概念用业务方使用的词（中文项目允许少量音译，如 `dingdan` 是禁忌，但 `orderNo` 可以）
5. **长度匹配作用域** —— 局部变量可以短（`i`、`item`），公共 API 必须长且具体

## 二、命名风格

| 语言族 | 变量/函数 | 类/类型 | 常量 | 文件 |
|---|---|---|---|---|
| JS / TS / Java | `camelCase` | `PascalCase` | `SCREAMING_SNAKE_CASE` | 视框架约定（多为 kebab 或 PascalCase） |
| Python | `snake_case` | `PascalCase` | `SCREAMING_SNAKE_CASE` | `snake_case.py` |
| Go | `camelCase`（私有）`PascalCase`（公有） | `PascalCase` | `PascalCase` | `snake_case.go` |
| Rust | `snake_case` | `PascalCase` | `SCREAMING_SNAKE_CASE` | `snake_case.rs` |

## 三、命名禁忌

### 通用
- ❌ 单字母变量（除循环 `i/j/k`、坐标 `x/y/z`、错误 `e/err`）
- ❌ 序号变量（`a1`、`a2`、`data1`、`data2`）
- ❌ 含糊词（`info`、`data`、`item`、`tmp`、`val`、`obj`、`util`）作为唯一名词使用
- ❌ 拼音命名（`yonghu`、`dingdan`、`xiaoshou`）
- ❌ 否定式（`isNotReady`），改用 `isReady` 反向取反
- ❌ 与语言/框架关键字重名（`type`、`class`、`new`、`list`）

### 函数命名
- 动词开头：`getUser`、`calculatePrice`、`validateEmail`、`isValidEmail`
- 布尔返回值用 `is/has/can/should` 前缀：`isActive`、`hasPermission`、`canDelete`
- 副作用大的函数显式说明：`saveAndSendEmail` 而不是 `saveUser`

### 类/类型命名
- 名词：`User`、`OrderService`、`PaymentGateway`
- 后缀表达职责：`UserService`、`OrderRepository`、`AuthMiddleware`、`PriceCalculator`
- 避免 `Manager`、`Helper`、`Util` 等模糊后缀，逼自己想出更具体的职责名

### 布尔变量
- 不写成 `flag`、`enabled`、`status`（含义太泛）
- 写清楚意图：`emailVerified`、`paymentReceived`、`profilePictureUploaded`

### 集合命名
- 复数：`users`、`orderItems`
- 类型暗示：`userMap`（键值对）、`userIds`（仅 id）、`userById`（按 id 索引）

### 常量命名
- 写出"为什么是这个值"：`MAX_LOGIN_ATTEMPTS_PER_HOUR = 5` 比 `MAX_RETRY = 5` 好
- 单位编进名字：`TIMEOUT_SECONDS = 30`、`MAX_FILE_SIZE_MB = 10`

## 四、目录与文件结构

- **按业务划分而不是按技术划分**（推荐）：
  - ✅ `features/user/`、`features/order/`、`features/payment/`
  - ❌ `controllers/`、`services/`、`models/`（小项目可以，大了就乱）
- **文件名表达内容**：`user-service.ts` 而不是 `index.ts`（除非项目约定 index 是入口）
- **测试文件就近**：`user-service.ts` 旁放 `user-service.test.ts`

## 五、代码风格硬性约束

### 长度
- 单文件 ≤ 500 行（注释除外）；超过考虑拆分
- 单函数 ≤ 50 行；超过要么真复杂（注释解释），要么该拆
- 单行 ≤ 120 字符（按项目约定，可调）

### 嵌套
- 嵌套深度 ≤ 4 层；超过用 **early return** 或抽出辅助函数压平
- 三元运算嵌套 ≤ 1 层；多了用 `if/else`

### 格式
- 缩进风格按项目统一（多数项目用 2 空格；Python/Go 例外）
- **强制使用 formatter**（Prettier / Black / gofmt / rustfmt）——不要手动对齐
- 强制使用 linter，CI 阻断未通过的代码

### 注释（极其重要）
- **默认不写注释**——好的命名 + 直白的代码 + 测试才是文档
- 只在"代码无法表达的 **为什么**"时写：
  - 业务约束（"按法务要求，金额展示精度必须到分"）
  - 非显然的算法选择（"用 X 算法是因为输入规模一定 < 1000"）
  - 临时妥协与 TODO（"临时绕过 X，长期方案见 ISSUE-123"）
- **禁止**：
  - 复述代码（`// 增加 1` 在 `i++` 旁边）
  - 注释掉的代码（删掉，git 里有）
  - 过期的 TODO（写明负责人或删除）
  - "本次任务的修改记录"（属于 commit message）

### 副作用与纯函数
- 优先写纯函数（输入→输出，无副作用）
- 有副作用的函数集中放置，命名要让人警惕（`saveX`、`sendY`、`updateZ`）

## 六、常见反例对照表

| 反例 | 改进 | 原因 |
|---|---|---|
| `function process(data)` | `function calculateMonthlyBill(usage)` | "处理什么"具体说 |
| `let flag = true` | `let isEmailVerified = true` | 表达意图 |
| `const a = 86400` | `const SECONDS_PER_DAY = 86400` | 解释魔法值 |
| `getUserData()` | `getUserProfile()` 或 `getUserOrders()` | data 太泛 |
| `// 设置标志位为 true`<br>`flag = true` | （删除注释，把 flag 改名） | 注释掩盖了真正问题 |
| `if (user.role === 1)` | `if (user.role === Role.ADMIN)` | 别用魔法数 |
| 函数嵌套 6 层 if | early return / 提取辅助函数 | 嵌套太深无法阅读 |
