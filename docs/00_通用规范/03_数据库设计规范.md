# 03 · 数据库设计规范

> 数据是项目最稳定也最难变更的部分。设计上的一个偷懒，会在三个月后让所有人都付出代价。

---

## 一、设计前置原则

1. **业务先于范式** —— 不要为了符合"第三范式"切碎表，导致每次查询 join 5 张表
2. **可演进 > 完美** —— 字段可以加，表可以扩；但表结构改动代价大，初始设计就考虑未来
3. **单一职责** —— 一张表只表达一个业务概念
4. **数据完整性靠数据库自己保证** —— 唯一性、外键、非空、check 约束尽量下沉到 DB 层，不要全靠应用层

---

## 二、命名规范

### 表名
- `snake_case` 单数：`user_account`、`order_item`、`payment_record`
- 模块前缀（可选）：`sys_user`、`crm_customer`、`fin_invoice`——便于跨表搜索定位
- 复数 / 单数项目内统一选一种，**不要混用**
- 禁止用保留字（`user`、`order` 在某些 DB 是保留字，加前缀或换名）

### 字段名
- `snake_case`：`user_id`、`created_at`、`is_deleted`
- 布尔字段加 `is_/has_` 前缀：`is_active`、`has_paid`
- 时间字段以 `_at` 结尾（时刻点）或 `_date` 结尾（日期）：`created_at`、`expire_date`
- 外键字段：`<table_name>_id`：`user_id`、`order_id`

### 索引名
- 主键：`pk_<table>`
- 唯一索引：`uk_<table>_<columns>`，如 `uk_user_email`
- 普通索引：`idx_<table>_<columns>`，如 `idx_order_user_id_created_at`

---

## 三、必备字段（每张业务表强烈建议都有）

| 字段名 | 类型 | 说明 |
|---|---|---|
| `id` | 主键（BIGINT / UUID / 雪花 ID） | 全项目统一一种主键策略 |
| `created_at` | TIMESTAMP | 创建时间，默认 CURRENT_TIMESTAMP |
| `updated_at` | TIMESTAMP | 更新时间，触发器或 ORM 自动维护 |
| `created_by` | 关联用户表的 ID | 谁创建的 |
| `updated_by` | 关联用户表的 ID | 谁最后改的 |
| `is_deleted` | BOOLEAN | 软删除标记，默认 false |
| `version` | INT | 乐观锁版本号（按需） |

### 软删除约定
- 所有查询**必须**带 `WHERE is_deleted = false`
- 建议建立视图或共享 SQL 片段统一过滤，避免遗漏
- 真删数据极少见，绝大多数业务用软删除
- 软删除字段建索引（特别是与其他查询条件组合时）

---

## 四、字段类型选择

| 业务类型 | 推荐 DB 类型（参考 PostgreSQL/MySQL） |
|---|---|
| 主键 ID | BIGINT（自增）/ UUID / VARCHAR(32)（雪花） |
| 用户名、邮箱、电话 | VARCHAR(255) |
| 状态枚举 | VARCHAR(32) 存字符串值（可读性 > 整数枚举） |
| 金额 | DECIMAL(18,4) 或 BIGINT 存"分"（**禁止 FLOAT/DOUBLE**） |
| 长文本 | TEXT，按需考虑全文索引 |
| 标志位 | BOOLEAN |
| 时间戳 | TIMESTAMP WITH TIME ZONE（PG）/ DATETIME（MySQL） |
| 日期（无时间） | DATE |
| JSON 数据 | JSONB（PG）/ JSON（MySQL，慎用） |

### 红线
- ❌ **禁止 FLOAT/DOUBLE 存金额** —— 精度丢失，永远的痛
- ❌ 状态码用 1/2/3 而不写枚举字典 —— 半年后没人记得 3 是什么
- ❌ TEXT 不带长度约束就上线 —— 业务字段必须有合理长度上限
- ❌ 把 JSON 当万能字段，应该建表的字段塞 JSON

---

## 五、索引

### 何时建
- WHERE / JOIN / ORDER BY / GROUP BY 频繁出现的字段
- 唯一性约束的字段（业务唯一键必须有唯一索引）
- 外键字段
- 时间范围查询（如订单按 `created_at` 查）

### 何时不建
- 区分度极低（如布尔字段单独建索引意义不大，除非配合其他字段做组合索引）
- 数据量小（< 1 万）的表

### 复合索引
- 顺序敏感：`(a, b)` 与 `(b, a)` 是不同索引
- 把**等值查询字段**放前面，**范围查询字段**放后面
- 例：`WHERE user_id = ? AND created_at > ?` → 索引 `(user_id, created_at)`

### 自检
- [ ] 频繁查询的字段有索引
- [ ] 业务唯一性有唯一索引
- [ ] 软删除字段在大表上有索引
- [ ] 没有重复/冗余索引（`(a)` 与 `(a, b)` 同时存在则 `(a)` 多余）

---

## 六、外键约束

- **强烈建议**保留外键约束（PG/MySQL 都支持）—— 数据完整性自动保证
- 外键的 `ON DELETE` 策略明确：`CASCADE` / `SET NULL` / `RESTRICT` —— **每个外键都要做选择**
- 极大表（千万级以上）可考虑去掉外键改用应用层校验，**前提是有强测试覆盖**

---

## 七、数据库迁移

### 工具
- 使用迁移工具（Flyway / Liquibase / Alembic / Prisma Migrate / Knex）
- 每个变更一个迁移文件，**不允许直接连生产 DB 改结构**
- 迁移文件**只增不改** —— 已发布的迁移不能改，要修就加新迁移

### 文件命名
- `V20260513001__create_user_table.sql` 或工具默认风格
- 文件内容**只描述本次变更**，幂等友好（用 `IF NOT EXISTS` 等）

### 危险操作的安全做法
| 操作 | 危险 | 安全做法 |
|---|---|---|
| 加 NOT NULL 字段到非空表 | 老数据违反约束 | 1）加可空字段 → 2）回填数据 → 3）改为 NOT NULL |
| 改字段类型 | 锁表、可能截断数据 | 1）加新字段 → 2）双写 → 3）回填 → 4）切读 → 5）删旧字段（多次发版） |
| 删字段 | 老代码会爆 | 1）代码停止写入 → 2）一段时间观察 → 3）下个迭代删 |
| 加索引（大表） | 长时间锁 | 用 `CREATE INDEX CONCURRENTLY`（PG）或 `pt-online-schema-change`（MySQL） |
| 删表 | 不可逆 | 极少做。先重命名（`_archived_`）观察一周再删 |

### 红线
- ❌ 迁移文件包含 `DROP` 时不写注释说明为什么
- ❌ 迁移文件包含数据修复 SQL 但没 LIMIT/事务保护
- ❌ 上线前没在等价数据量的库上演练过迁移

## 八、查询性能

### 必做
- 所有查询经过 explain 分析（特别是涉及 join、子查询、IN 大列表的）
- 慢查询日志开启（如 PG `log_min_duration_statement = 200ms`）
- ORM 警惕 N+1：列表查询时关联数据一次性 join 或 IN 查询，**不要循环查**
- 大数据量遍历用流式或分页，**禁止一次性 SELECT 全表到内存**

### 禁忌
- ❌ `SELECT *`（生产代码）—— 列出明确字段
- ❌ 在 WHERE 字段上做函数变换（如 `WHERE DATE(created_at) = ?`）—— 索引失效
- ❌ LIKE 前置通配（`'%xxx'`）—— 全表扫
- ❌ OR 条件分别命中不同索引但合在一起无法走索引

---

## 九、事务与并发

- 跨表写入用事务，事务尽量短
- **禁止在事务中调用外部 HTTP/RPC/MQ** —— 连接被长期占用
- 高并发场景用乐观锁（`version` 字段）或唯一约束保证一次成功
- 业务唯一性靠唯一索引兜底，**不要只靠应用层 "查后插"**
- 详见 `05_后端开发规范.md`

---

## 十、数据安全与脱敏

- 密码必须 hash（bcrypt / argon2），**禁止明文或可逆加密**存储
- 身份证、手机号、邮箱等 PII 数据按合规要求：考虑加密存储 + 日志脱敏 + 接口脱敏
- 测试库数据必须脱敏后才能使用生产数据 dump
- 备份保留策略、删除/导出权限走审计

---

## 十一、字典与枚举管理

- 业务枚举值（订单状态、用户角色等）记录在 `docs/11_字典/系统字典.md`
- 数据库列里**存字符串值**而不是数字（提高可读性）
- 枚举变更：新增允许，**删除/重命名需要谨慎处理老数据**

---

## 十二、自检清单

- [ ] 表名、字段名符合命名规范
- [ ] 必备字段（id、created_at、updated_at、is_deleted）齐全
- [ ] 主键策略与项目统一
- [ ] 金额用 DECIMAL 或整数，不用 FLOAT
- [ ] 时间字段类型与时区策略一致
- [ ] 业务唯一性有唯一索引保护
- [ ] 常用查询字段已建索引
- [ ] 软删除字段在大表上已建索引
- [ ] 外键约束保留或明确放弃理由
- [ ] 字典/枚举已记录到字典文档
- [ ] 迁移文件用了安全步骤（特别是 ALTER 大表）
