# 05 · 后端开发规范

> 后端是数据的守门人。一行错误的逻辑，可能是一万条错误的数据。

---

## 一、总原则

1. **分层清晰** —— 不同层做不同事，依赖方向单一
2. **薄控制器、厚服务、纯仓储** —— 业务逻辑集中在服务层，便于测试
3. **错误透明** —— 不吞异常，统一错误响应
4. **可观测** —— 日志、追踪、指标三件套
5. **可测试** —— 业务逻辑不依赖具体框架/网络

---

## 二、分层架构（通用模型）

```
┌────────────────────────────────────────┐
│  Controller / Handler  (HTTP 边界)     │  接参、校验、调服务、返响应
├────────────────────────────────────────┤
│  Service / UseCase     (业务逻辑)      │  业务规则、事务编排、调用仓储/外部
├────────────────────────────────────────┤
│  Repository / DAO      (数据访问)      │  增删改查，纯数据，无业务
├────────────────────────────────────────┤
│  Domain / Entity       (领域模型)      │  实体、值对象、领域规则
└────────────────────────────────────────┘
       External Adapter (MQ / HTTP / Cache / Storage / ...)
```

### 依赖方向
- Controller → Service → Repository
- **禁止反向依赖**：Service 不能调 Controller，Repository 不能调 Service
- **禁止跨层调用**：Controller 不能直接调 Repository

---

## 三、Controller 层（HTTP 边界）

### 做什么
- 接收请求参数
- 参数格式校验（类型、必填、长度、格式）
- 调用 Service
- 转换 Service 返回为 HTTP 响应
- 错误转换为 HTTP 状态码

### 不做什么
- ❌ 业务规则判断（"如果用户已支付则..."）—— 属于 Service
- ❌ 直接拼 SQL —— 属于 Repository
- ❌ 调用其他外部服务 —— 属于 Service / Adapter

### 控制器代码风格
- 单方法行数 ≤ 30 行（除大量字段映射）
- 业务异常按规范错误码抛出，**不要硬编码字符串**
- 鉴权 / 权限拦截通过中间件/装饰器，不要写在方法里
- 请求/响应 DTO 与领域模型分开，**禁止把数据库实体直接返给前端**

---

## 四、Service 层（业务核心）

### 做什么
- 业务规则判断与编排
- 事务管理
- 调用 Repository 操作数据
- 调用外部服务（MQ / RPC / HTTP / 第三方 API）
- 抛出业务异常（带明确 code）

### 设计原则
- 一个 Service 方法对应一个业务用例
- 方法命名表达业务意图：`registerUser`、`cancelOrder`、`refundPayment`
- 参数尽量用 DTO/Command 对象而不是一堆散参数
- **业务规则不要散落** —— 同样的判断（如"是否可作废"）只写一处，多处复用

### 事务规则（极重要）
- `@Transactional` / 事务装饰器加在 Service 方法上
- **禁止**：
  - 在事务内调用外部 HTTP/RPC（连接被长期占用，可能死锁）
  - 在事务内发 MQ 消息（应用 Outbox 模式）
  - 用 try-catch 吃掉异常导致事务不回滚
  - 框架自调用导致事务失效（`this.xxx()` 不走代理）
- 事务范围尽量小，**只包必要的 DB 操作**

---

## 五、Repository 层（数据访问）

### 做什么
- CRUD 操作
- 复杂查询的 SQL / ORM 表达
- 数据库特定的优化（索引提示、分页）

### 不做什么
- ❌ 业务判断（不要写 `if user.status == 'active'` 然后做不同查询）
- ❌ 直接抛业务异常（抛技术异常，由 Service 转译）

### SQL 规范
- 参数化查询，**禁止字符串拼接**
- `SELECT *` 仅限单元测试 / 调试，生产代码列明字段
- 涉及大数据量的查询必须考虑：分页、索引、limit、流式
- 软删除：所有查询带 `WHERE is_deleted = false`，**用共享片段/方言/视图避免遗漏**
- 多表 join 不超过 4 表，超过考虑反范式或拆查询

---

## 六、领域模型 / 实体

- 实体保持纯净，不依赖框架（便于单元测试）
- 业务规则可放在实体方法中（如 `Order.canCancel()`）
- DTO（数据传输）与 Entity（数据库实体）分开
- 值对象（VO）用于无 ID 的概念：`Money`、`Address`、`PhoneNumber`

---

## 七、外部依赖（HTTP / MQ / Cache / 文件）

### HTTP 调用
- 必有超时（连接超时 + 读超时）
- 必有重试策略（**只对幂等接口重试**）
- 必有熔断降级（Circuit Breaker）
- **不在事务中调用**

### 消息队列
- **消费者必须幂等**（按业务唯一键去重）
- 失败有重试 + 死信队列
- 发送消息建议用 Outbox 模式（与业务数据同事务写一张消息表，独立任务投递），避免事务和 MQ 不一致

### 缓存
- 明确**失效策略**：定时失效 / 写时失效 / 双写
- **缓存与 DB 双写不一致**是常见坑：先更 DB 再删缓存（Cache-Aside），不要先更缓存
- 缓存键命名：`<业务>:<实体>:<id>` 如 `user:profile:12345`
- 缓存大对象要警惕：考虑压缩或拆分

### 文件存储
- 上传校验：类型、大小、内容（不要只信扩展名）
- 路径用对象存储（S3 / OSS / MinIO）而不是服务器本地盘
- 文件 URL 鉴权（签名 URL / 鉴权代理）

---

## 八、并发与一致性

### 写入冲突
- 乐观锁：实体加 `version` 字段，UPDATE 时带 WHERE version=?
- 唯一索引兜底业务唯一性
- 高并发热点（库存、计数）用分布式锁 / Redis 原子操作 / 消息队列单线程消费

### 幂等性
- 创建类接口接受 `Idempotency-Key`，记录已处理过的请求 ID
- 状态机推进操作只在合法前置状态下生效（`WHERE status = 'pending'`）

### 分布式事务
- 优先用**最终一致性 + 消息**而不是 2PC
- 业务做好补偿机制（Saga 模式）

---

## 九、日志（可观测性的核心）

### 等级
| 等级 | 用途 |
|---|---|
| DEBUG | 开发调试，生产关闭或采样 |
| INFO | 关键业务事件（订单创建、支付完成、状态流转） |
| WARN | 异常但可恢复（重试成功、降级生效、限流） |
| ERROR | 业务/系统错误，必须有完整堆栈和上下文 |

### 必含字段
- `traceId` / `requestId`（用于关联请求链）
- `userId`（关键操作必须有）
- 业务实体 ID（订单号、单据号）
- 关键参数（脱敏后）

### 红线
- ❌ 打印明文密码、token、身份证、手机号、银行卡
- ❌ 在循环里大量 INFO 日志（撑爆磁盘）
- ❌ ERROR 只打消息没有堆栈
- ❌ 日志只 print 给自己看 —— 上 ELK / Loki / 云日志服务

详见 `07_错误处理与日志规范.md`。

## 十、错误处理

### 抛错
- 业务错误：用项目统一的业务异常类（带 code、message、字段错误）
- 技术错误：让框架/底层抛出，**不要无意义包一层**
- **禁止吞异常**：`catch` 后必须 log + 抛出（或显式声明吞掉的理由）

### 统一错误响应
- 全局异常处理器（Spring `@ControllerAdvice` / Express middleware / FastAPI exception handler）
- 业务异常 → 4xx + 业务错误码
- 未预期异常 → 5xx + traceId + 隐藏堆栈（堆栈只入日志，不返前端）

详见 `07_错误处理与日志规范.md`。

---

## 十一、配置管理

- 配置分环境：本地、开发、测试、预发布、生产
- 敏感配置（DB 密码、API key）**不入代码仓库**，用环境变量 / 配置中心 / Vault
- 配置变更可追溯（配置中心审计 / Git 历史）
- 启动时校验关键配置（缺失则 fail-fast，不要带病启动）

---

## 十二、API 设计

详见 `02_API接口设计规范.md`。

后端开发者特别注意：
- 接口与文档同步更新
- 字段命名遵循统一风格
- DTO 不要直接复用 Entity
- 列表/分页响应结构全局统一

---

## 十三、安全

详见 `06_安全规范.md`。

后端开发者特别注意：
- 所有接口默认鉴权
- 数据权限在 SQL 层
- 输入校验在后端做（不能信任前端）
- SQL 注入：参数化；ORM 不要拼字符串
- 文件上传校验内容、限大小、隔离存储

---

## 十四、性能

### 必做
- 关键查询打 EXPLAIN，避免全表扫
- 大表查询有索引覆盖
- 列表 / 导出避免 N+1（一次 join / IN 查询）
- 缓存热点数据
- 异步处理耗时操作（导出报表、批量任务）走队列

### 监控
- 接口 P95 / P99 耗时
- DB 慢查询
- 缓存命中率
- 队列堆积情况
- 错误率与 5xx 比例

---

## 十五、测试

- 业务逻辑层（Service）必须有单元测试
- Repository 用真实数据库容器（Testcontainers）测试，**不用方言不同的内存 DB**
- Controller 用 MockMvc / supertest 等做接口契约测试
- 详见 `104_测试工作流程.md`

---

## 十六、自检清单

- [ ] 控制器薄，无业务逻辑
- [ ] Service 方法对应一个业务用例
- [ ] 事务范围最小，无外部调用卷入
- [ ] Repository 纯 CRUD，无业务判断
- [ ] DTO 与 Entity 分离
- [ ] 所有接口有鉴权和权限校验
- [ ] 数据权限在 SQL 层过滤
- [ ] 输入参数后端校验
- [ ] 日志带 traceId、userId、业务 ID
- [ ] 错误响应统一格式
- [ ] 关键查询有索引
- [ ] 缓存失效策略明确
- [ ] 外部调用有超时和重试
- [ ] 消费者幂等
- [ ] Service 层有单元测试覆盖核心业务
