# 04 · 前端开发规范

> 前端是用户离体验最近的一层。一行写不好的代码，就是一千个用户的 1 秒不爽。

---

## 一、总原则

1. **先想交互再写代码** —— 加载态、空态、错误态、权限态四态齐全是底线
2. **数据流单向、状态尽量上提** —— 避免双向绑定地狱
3. **拥抱框架约定** —— 不要和框架打架
4. **可访问性是基本功** —— 键盘、焦点、ARIA 是底线
5. **视觉风格一致** —— 用项目设计系统的组件，不要自己造轮子

---

## 二、项目结构（推荐分层）

```
src/
├── api/                  接口调用层（与后端契约）
├── assets/               静态资源
├── components/           通用组件（业务无关）
├── features/             业务功能模块（推荐）
│   └── user/
│       ├── api.ts        本模块的接口
│       ├── components/   本模块专用组件
│       ├── store.ts      本模块状态
│       ├── types.ts      本模块类型
│       ├── pages/        本模块页面
│       └── index.ts      对外导出
├── hooks/                可复用 hooks / composables
├── stores/               全局状态
├── router/               路由配置
├── utils/                纯工具函数
├── types/                全局类型
└── styles/               全局样式
```

> **按业务划分而不是按技术划分**。`features/user/` 比 `controllers/` + `services/` + `models/` 散落各处更好维护。

---

## 三、组件设计

### 组件分类
- **页面组件**：路由直接渲染，承担数据获取、布局
- **业务组件**：处理具体业务（如 `UserForm`、`OrderList`），依赖业务概念
- **通用组件**：业务无关，纯展示/交互（如 `Button`、`Modal`、`Tabs`）

### 设计原则
- **单一职责**：一个组件一件事
- **可测试**：业务逻辑尽量从组件抽离到 hooks / composables / 纯函数
- **明确接口**：props 必须有类型；emits 必须声明
- **状态最小化**：能从 props 派生的不放 state；能从 url 派生的不放 state
- **避免深层 props 传递**（3 层以上）：用 context / 状态库代替

### 命名
- 文件名：`PascalCase`（多数框架约定）：`UserList.vue`、`OrderDetail.tsx`
- props 名：`camelCase`
- events / emits 名：`kebab-case`（HTML 友好）：`<MyComp @user-selected="..." />`

---

## 四、状态管理

### 选型
- 局部状态：组件内 state
- 跨组件浅层：props / events 或 context
- 复杂全局：状态库（Pinia / Redux / Zustand / Jotai 等）

### 红线
- ❌ 把所有数据都塞进全局 store —— 把组件搞成 props 透传中转站
- ❌ 在多处 fetch 同一份数据 —— 用 SWR / TanStack Query / 状态库统一
- ❌ store 里塞临时 UI 状态（模态框开关、loading）——这种属于组件内 state

### 服务端状态
- 推荐使用 SWR / TanStack Query / Vue Query 这类库统一管理：
  - 缓存
  - 自动重试
  - 失效与重新拉取
  - 加载/错误/空状态

---

## 五、四态齐全（基本功）

页面/数据驱动的组件**必须**显式处理四种状态：

| 状态 | 表现 |
|---|---|
| 加载态 | Skeleton / Spinner / 占位 |
| 空态 | 友好提示 + 引导操作（"暂无数据，点这里创建"） |
| 错误态 | 错误描述 + 重试按钮，**不要只显示空白页** |
| 权限态 | 无权限明确告知（"你无权查看此内容"） |

**禁止**：加载完成后页面空白无反馈、报错时只在控制台打印。

## 六、表单

### 必备
- **客户端校验**用于即时反馈：长度、格式、必填
- **服务端校验**是真校验：返回的错误必须能映射回字段
- 提交按钮 `loading` + 防重复提交（按钮 disable 或请求中标记）
- 错误信息显示在对应字段下方，**不要只 toast 一个总错误**
- 离开未保存表单时**给确认**（`beforeunload` / 路由守卫）

### 红线
- ❌ 提交后还能再点 → 数据重复
- ❌ 后端返回的字段级错误，前端却显示一个通用提示
- ❌ 大表单一次性渲染 200 个 input 不分组不分步骤

---

## 七、列表与表格

### 必备
- 分页 / 滚动加载（数据量 > 100 用虚拟滚动）
- 排序与筛选状态可分享（写入 URL query）
- 每行操作权限按当前用户角色控制
- 空态、加载态明确

### 表格规范
- 列宽：关键列固定，长文本截断 + hover 展示全文
- 对齐：数字右对齐，文本左对齐，状态居中
- 金额：千分位 + 等宽字体 + 右对齐
- 操作列：常用操作放外面，次要操作放更多菜单
- 大数据量：开虚拟滚动；导出**走后端**而不是前端拼接

---

## 八、路由

- 路由命名清晰、可读
- 路由参数 vs query：决定性的资源 ID 放 path（`/users/:id`），可选过滤放 query（`?status=active`）
- **路由守卫的执行顺序**要测试 —— 经典坑：菜单未加载完时跳转，导致 404 / 闪烁
- 懒加载页面级路由
- 未授权页面有明确跳转目标（登录页 / 403 页）

---

## 九、网络请求

### 封装
- **统一入口**：所有请求经一个 client（拦截器统一加 token、统一处理错误码）
- **类型化**：基于 OpenAPI 生成类型，**不要手抄字段**
- **错误处理**：
  - 业务错误（4xx）→ 在调用点处理或 toast
  - 网络错误 / 5xx → 拦截器统一 toast / 上报
  - 401 → 跳登录
  - 403 → 弹无权限提示
- **超时与取消**：长请求支持取消（AbortController），切页面自动取消

### 红线
- ❌ 把 token 拼接到 URL（应该在 Header）
- ❌ 错误响应直接 `JSON.stringify` 给用户看
- ❌ 同一份数据在多处独立请求多次（用缓存/Query 库）

---

## 十、样式与设计系统

### 优先级
1. 用项目设计系统的组件（`<Button>` / `<Card>` / `<Input>`...）
2. 用设计系统的 design token（颜色、间距、字号、阴影）
3. 才考虑自己写
- **禁止**业务页面里出现写死的颜色、字体、间距（如 `color: #ff0000`）

### CSS 组织
- 优先使用项目约定（Tailwind / CSS Modules / CSS-in-JS / SCSS 模块化）
- 避免全局样式污染：组件级样式使用 scope / modules / 嵌套类名
- 响应式：移动优先（min-width 媒体查询）

### 视觉一致性
- 不要让"AI 自动生成"的代码引入异样的视觉风格 —— 严格按设计系统
- 没有 UI 设计稿时**先停下来**，不要乱发挥（让产品/设计补稿）

---

## 十一、可访问性（A11y）

- 所有交互元素可键盘操作（Tab、Enter、Esc）
- 焦点可见（不要用 `outline: none` 而不补充焦点样式）
- 图片有 alt（装饰性图片用空 alt）
- 表单 label 与 input 关联
- 颜色对比度达标（WCAG AA：正常文字 4.5:1，大字 3:1）
- 关键操作有 aria-label 或可读文本，**不要只用 icon**

---

## 十二、性能

### 必做
- 图片懒加载、合适尺寸（不要让浏览器缩 10MB 图）
- 路由 / 组件懒加载
- 长列表虚拟滚动
- 防抖（搜索输入）、节流（滚动、resize）
- 大型第三方库按需引入（lodash / antd / element 等）

### 监控
- 关键页面打 LCP / FCP / CLS 埋点
- 接口耗时埋点
- JS 错误上报（Sentry / 自建）

---

## 十三、国际化（i18n）

如果项目支持多语言：
- 所有面向用户的文案必须走 i18n，**禁止硬编码中文/英文**
- key 命名按层级：`page.user.list.title`
- 日期、数字、货币用 i18n 库的 formatter

---

## 十四、构建与发布

- `build` 必须无 type error、无 warning
- 严格 TypeScript（strict: true）
- Lint 在 CI 阻断
- 单元测试在 CI 阻断
- 资源带 hash 防缓存
- Source map 生产环境慎重（部署到 Sentry 但不暴露给用户）

---

## 十五、自检清单

- [ ] 组件单一职责，业务逻辑抽离到 hooks / 纯函数
- [ ] props 有类型，emits 有声明
- [ ] 四态齐全：加载 / 空 / 错误 / 权限
- [ ] 表单提交防重复、错误回显到字段
- [ ] 列表分页/虚拟滚动；导出走后端
- [ ] 服务端状态用统一缓存方案，不重复请求
- [ ] 视觉风格走设计系统，无硬编码颜色字号
- [ ] 可访问性达标：键盘、焦点、对比度、aria
- [ ] 国际化 key 完整（如有 i18n）
- [ ] 浏览器 Console 干净：无红色错误、无未捕获 Promise rejection
- [ ] Network 面板正常路径无 4xx/5xx
