# 项目协作纲领（AI 必读）

> 本文件在每次会话开始时自动加载，是 AI 与开发者协作的最高纲领。
> 所有规范与工作流均按本文件路由，**违反纲领的行为视为开发未完成**。

---

## 0. 项目基本情况

> 复制本套件后，请在此处补充本项目的：技术栈、核心业务、运行环境、关键依赖。
> 信息越具体，AI 越不会乱猜。

- **技术栈**：（前端 / 后端 / 数据库 / 中间件 / 部署形态）
- **业务领域**：（一句话说明做什么）
- **代码主目录**：（前端代码位置 / 后端代码位置）
- **文档主目录**：`docs/`
- **运行/调试方式**：（启动命令、端口、本地依赖）

## 1. 三条铁律（不可违反）

1. **禁止猜测** —— 业务规则、字段含义、接口契约、数据库结构有不确定时，**先问，再做**。宁可多问一句，不可写错一行。
2. **改代码前先读规范** —— 凡涉及代码修改，必须先查阅 `docs/00_通用规范/` 中对应规范文档，按规范实现；规范缺失或冲突时，**先与开发者对齐再动手**。
3. **方案需开发者确认才可执行** —— 任何"动代码"的任务，AI 须先输出方案/影响范围/风险，**得到开发者明确"确认"或"开始"字样后**才进入开发。这条比一切效率诉求都重要。

## 4. 交付物归档约定

工作流产出的文档必须落盘到固定位置，便于后续检索与回归。**禁止只在对话中口头说明而不归档**。

| 产出物 | 归档位置 | 命名 |
|---|---|---|
| 需求分析文档 | `docs/01_需求分析/` | `yyyymmdd_xxx需求分析.md` |
| 实施/优化方案设计 | `docs/02_功能设计/` | `xx功能_实施方案_yyyymmdd.md` |
| BUG 记录 | `docs/03_问题记录/` | `yyyymmdd_BUG_xxx.md` |
| 用户手册 | `docs/10_用户手册/` | `yyyymmdd_xxx功能说明.md` |
| 代码审查报告 | `docs/04_代码审查/` | `yyyymmdd_审查_xxx.md` |
| 测试用例 / 回归用例 | 写入对应方案/BUG 文档末尾，回归用例同时拷贝到 `docs/05_回归用例/` 永久保留 |

模板见 [`docs/20_模板/`](docs/20_模板/)。

## 5. 强制的"工作完成"定义（DoD）

任何工作流自称"完成"前，**逐条**对照 [`docs/90_最佳实践/交付前自检清单.md`](docs/90_最佳实践/交付前自检清单.md)，缺项 = 未完成。

简版速记：
- [ ] 方案已与开发者确认
- [ ] 代码符合相关规范
- [ ] 已自审一遍（错误 / 业务一致 / 回归影响）
- [ ] 关联文档已同步（需求 / 方案 / 用户手册 / 字典 / 菜单权限说明）
- [ ] 测试用例已设计并通过（按 104 工作流）
- [ ] PR/提交描述写清"做了什么、为什么、怎么测"

## 6. 沟通与节奏

- **方案先行**：超过 10 分钟工作量的任务，先发方案再开工
- **进度可见**：每完成一个子步骤主动汇报，让开发者随时可以喊停
- **失败要响**：报错、卡点、与预期不符，**立即抛出**，不要自行绕路
- **不修无关代码**：本次任务范围之外的代码，发现问题只记录到 `docs/03_问题记录/` 待后续处理，**不顺手改**

---

## 7. 重要避坑提醒

在动手前**默念一遍** [`docs/90_最佳实践/通用陷阱清单.md`](docs/90_最佳实践/通用陷阱清单.md)，对照本次任务自问：这些坑我会不会重蹈。

特别警惕：
- 日期开闭区间不一致 / 时区混乱
- 金额精度丢失 / 百分比与小数双重转换
- 软删除字段忘记过滤
- 越权（功能权限 OK 但数据权限缺失 / 导出绕过列表过滤）
- 事务边界错误（@Transactional 失效、外部调用卷入事务）
- 缓存与 DB 双写不一致
- 改了一处忘了改相似代码（grep 没做全）
